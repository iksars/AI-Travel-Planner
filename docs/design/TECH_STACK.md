# AI 旅行规划师 - 快速原型技术方案 (V1 - 已确认)

**1. 核心指导思想：为速度而生，为验证核心价值**

本方案是为在最短时间内（1-2周）构建一个可工作的“AI旅行规划师”原型而设计。所有决策都以**开发速度**为唯一衡量标准。我们将在产品核心价值（AI生成计划 + 地图交互）得到市场验证后，对整个系统进行重构。

**2. 架构选型：单体架构 (Monolith)**

*   **决策：** 所有前后端代码将存在于一个单一的代码仓库和应用中。
*   **理由：** 极速开发，零API联调成本，最简单的部署模型，非常适合单人或小团队冲刺。

**3. 技术栈 (The "Fast Stack")**

| 领域 | 选型技术 | 决策理由（为什么它最快？） |
| :--- | :--- | :--- |
| **后端框架** | **Python + Django** | **自带强大的Admin后台**，可零成本、零代码实现PRD中所有的后台管理功能（用户管理、内容配置）。其成熟的ORM和认证系统能极大加速开发。 |
| **前端框架** | **Vue.js + Ant Design Vue** | **拥有现成的高质量UI组件库**。UX/UI规格中要求的复杂地图交互、卡片列表、图表等，可通过现有组件快速拼装，避免手写CSS和复杂的交互逻辑，聚焦于核心功能实现。 |
| **数据库** | **SQLite** | **零配置，文件即数据库**。完美满足原型功能测试，无需安装、配置或管理独立的数据库服务器，项目启动即用。 |
| **核心AI能力** | **直接调用第三方API** | **采购成熟服务，不自研**。行程规划将通过调用大语言模型API（如OpenAI, Google Gemini等）实现，语音识别功能在原型阶段被简化。 |

**4. 部署方案 (The "Good Enough" Deployment)**

*   **首选方案 (在线测试):** **PaaS平台 (Render / Heroku)**。通过关联GitHub仓库，实现`git push`自动构建和部署。这为团队和早期用户提供了公开的测试URL，且运维成本为零。
*   **备选方案 (本地演示):** 提供一个`docker-compose.yml`文件。在任何安装了Docker的机器上运行`docker-compose up`即可一键启动整个应用，用于稳定快速的本地演示。

**5. 已确认的简化与牺牲**

为了聚焦于核心功能，我们明确在原型阶段进行以下简化和牺牲：

*   **后台管理 (Admin):** **确认100%依赖Django Admin**。所有后台操作（如查看用户、管理内容）均通过Django自带的、未经美化的后台界面手动完成。**不开发**任何独立的前端后台管理页面。
*   **费用记录 (Expense Tracking):** **确认简化为“手动文本记账”**。暂时移除语音识别记账功能，以规避API联调和前端语音动效的开发时间。
*   **用户认证 (Authentication):** **确认只实现基础的“邮箱/用户名+密码”登录**。移除所有第三方社交登录（如微信、Google）的集成，以减少外部依赖和开发复杂度。
*   **地图交互 (Map Interaction):** **确认保留完整的地图交互功能**。地图与行程列表的联动（高亮、缩放、聚焦）是核心用户体验，**此部分不做简化**。
*   **其他牺牲：**
    *   **可扩展性：** 单体架构和SQLite数据库无法支持高并发。
    *   **代码质量：** 为了速度，将存在硬编码和高耦合，缺少测试。
    *   **安全性：** 仅做基础认证，忽略所有高级安全措施。

这份文档将作为我们开发工作的“唯一真相来源”。我们现在可以开始搭建项目结构了。
